The lower levels

What’s actually happening down there

Talking to a terminal

  • Terminals speak VT/ANSI escape sequences
    ESC[31m → red, ESC[2J → clear screen…​

  • Input: key events, mouse events, resize signals

  • Different on every OS and terminal emulator 🙃

That’s what the backends abstract away.

The backend contract

// All three backends implement the same Terminal interface
try (var terminal = Terminal.create()) {   // picks best available
    terminal.enterAlternateScreen();
    terminal.enableRawMode();
    terminal.enableMouseCapture();
    // ...
    terminal.exitAlternateScreen();
}

AutoCloseable. Works with try-with-resources. Very Java.

Core types: Buffer, Cell, Rect

// A Buffer is a 2D grid of Cells
var buffer = Buffer.empty(Rect.of(0, 0, 80, 24));

// A Cell has a symbol and a Style
buffer.set(0, 0, Cell.of("H", Style.builder().fg(Color.RED).bold().build()));

// Rect: x, y, width, height — used everywhere for layout
var area = Rect.of(10, 5, 60, 14);

Diff-based rendering: only changed cells are sent to the terminal.

Layout: constraints

var layout = Layout.horizontal(
    Constraint.percentage(30),   // sidebar
    Constraint.min(20),          // main content, at least 20 cols
    Constraint.length(10)        // fixed-width panel
);

Rect[] rects = layout.split(frame.area());

Flexbox-inspired. Responsive. No CSS headaches.

Immediate Mode: the raw loop

try (var tui = ImmediateTui.create()) {
    tui.run(frame -> {
        var para = Paragraph.builder()
            .text(Text.from("Hello, TamboUI!"))
            .block(Block.builder().borders(Borders.ALL).build())
            .build();
        frame.renderWidget(para, frame.area());
    });
}

Every frame: clear, re-render everything, diff, push to terminal.

Widgets

  • Paragraph — styled text with wrapping

  • List — scrollable item lists

  • Table — tabular data with column constraints

  • Gauge / LineGauge — progress bars

  • Chart — sparklines, bar charts

  • Canvas — pixel-level drawing

  • Image — yes, actual images in the terminal

  • Calendar, Spinner, and more…​

The higher levels

Where it gets fun

TuiRunner: stop writing boilerplate

try (var tui = TuiRunner.create()) {
    tui.run(
        (event, runner) -> switch (event) {
            case KeyEvent k when k.isQuit() -> { runner.quit(); yield false; }
            default -> false;
        },
        frame -> {
            frame.renderWidget(myWidget, frame.area());
        }
    );
}

Event loop, terminal setup, redraws — all handled.
You write: event handling and rendering. That’s it.

Mouse + animation

var config = TuiConfig.builder()
    .mouseCapture(true)
    .tickRate(Duration.ofMillis(16))  // ~60fps
    .build();

try (var tui = TuiRunner.create(config)) {
    tui.run(
        (event, runner) -> switch (event) {
            case MouseEvent m -> { handleMouse(m); yield true; }
            case TickEvent t  -> { updateAnimation(t); yield true; }
            // ...
        },
        frame -> render(frame)
    );
}

The Toolkit DSL

try (var runner = ToolkitRunner.create()) {
    runner.run(() ->
        vBox(
            markupText("[bold]My App[/bold]"),
            hBox(
                sidebar(),
                mainContent()
            ),
            statusBar()
        )
    );
}

Declarative. Composable. Automatic focus management and event routing.

CSS in the terminal

/* app.tcss */
.sidebar {
    width: 30%;
    border: solid green;
}

.selected {
    background: #005fd7;
    color: white;
    text-style: bold;
}
runner.loadStylesheet("app.tcss");
runner.switchTheme("dark");  // at runtime!

Separate style from logic. Switch themes live. Your designer will thank you.

GraalVM native

# Build a native executable — no JVM required to run
./gradlew nativeCompile

# Result: a ~10MB binary, starts in milliseconds
./build/native/nativeCompile/myapp

TamboUI ships with all the GraalVM metadata out of the box.
Zero config for reflection, serialization, resources.
