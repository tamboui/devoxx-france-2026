=== TuiRunner: what you stop writing

* Poll / parse / dispatch event loop
* Resize → debounced redraw (no flicker on drag)
* Tick events for animation, disabled when you don't need them
* Thread-safe `runOnRenderThread` + a managed scheduler
* Pluggable `RenderErrorHandler`: crashes don't wreck the terminal

[.small]
The pieces every TUI framework rewrites badly. You get them once, right.

=== Toolkit: the next step up

With TuiRunner you still render frame-by-frame. +
Toolkit lets you describe the UI from state, and handles the rest:

* You return a tree of `Element` from your state; the runner diffs it
* Automatic **focus management** (Tab / Shift+Tab)
* Automatic **event routing**: events go to the focused element
* `@OnAction` handlers: no manual key wiring
* CSS styling with `:focus`, `:hover` pseudo-classes
* Hot theme switching at runtime

=== CSS in the terminal

[source,css]
----
/* app.tcss */
.sidebar        { width: 30%; border: solid green; }

#nav:focus      { border-type: double; border-color: cyan; }
#nav:hover      { background: #1e2838; }

.item:selected  { background: #005fd7; color: white; text-style: bold; }
----

[source,java]
----
runner.loadStylesheet("app.tcss");
runner.switchTheme("dark");   // at runtime, live!
----

Pseudo-classes. Themes. No recompile. +
Your designer can edit a `.tcss` file while the app is running.

=== JFR tracing, because it's Java

[source,bash]
----
jcmd <pid> JFR.start name=tamboui \
    settings=profile duration=60s filename=tamboui.jfr
----

Events: `dev.tamboui.terminal.draw`, `dev.tamboui.toolkit.route`, +
focus changes, drag state, event routing… +
Profile your TUI like any other Java app. 🔥
