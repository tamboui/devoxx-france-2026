=== TuiRunner: what you stop writing

* Poll / parse / dispatch event loop
* Resize → debounced redraw (no flicker on drag)
* Tick events for animation, disabled when you don't need them
* Thread-safe `runOnRenderThread` + a managed scheduler
* Pluggable `RenderErrorHandler`: crashes don't wreck the terminal

[.small]
The pieces every TUI framework rewrites badly. You get them once, right.

=== TuiRunner: the whole app

[source,java]
----
int[] count = {0};
try (var tui = TuiRunner.create()) {
    tui.run(
        (event, runner) -> switch (event) {
            case KeyEvent k when k.isQuit() -> { runner.quit(); yield false; }
            case KeyEvent k when k.isUp()   -> { count[0]++; yield true; }
            case KeyEvent k when k.isDown() -> { count[0]--; yield true; }
            default -> false;
        },
        frame -> frame.renderWidget(
            Paragraph.of("Count: " + count[0]).centered(),
            frame.area())
    );
}
----

A complete, resizable, crash-safe counter app. ~15 lines of body.

=== TuiRunner: config knobs

[source,java]
----
var config = TuiConfig.builder()
    .mouseCapture(true)
    .tickRate(Duration.ofMillis(16))        // ~60 fps
    .resizeGracePeriod(Duration.ofMillis(250))
    .bindings(BindingSets.vim())            // hjkl out of the box
    .errorHandler(RenderErrorHandlers.displayAndQuit())
    .build();

try (var tui = TuiRunner.create(config)) {
    tui.scheduleRepeating(Duration.ofSeconds(1), this::pollBackend);
    tui.run(this::onEvent, this::render);
}
----

`scheduleRepeating` runs on the scheduler thread. +
Use `tui.runOnRenderThread(...)` to mutate UI state safely.

=== TuiRunner: error handling, for real

[source,java]
----
TuiConfig.builder()
    .errorHandler(RenderErrorHandlers.displayAndQuit())
    .build();
----

Exception in a widget? The framework catches it, renders a
*scrollable stack trace inside your TUI*, and exits cleanly. +
No mysterious "terminal stuck in raw mode" after a crash.

=== Toolkit: the next step up

With TuiRunner you still render frame-by-frame. +
Toolkit lets you describe the UI from state, and handles the rest:

* You return a tree of `Element` from your state; the runner diffs it
* Automatic **focus management** (Tab / Shift+Tab)
* Automatic **event routing**: events go to the focused element
* `@OnAction` handlers: no manual key wiring
* CSS styling with `:focus`, `:hover` pseudo-classes
* Hot theme switching at runtime

=== Toolkit: a complete app

[source,java]
----
public class CounterApp extends ToolkitApp {
    private int count = 0;

    @Override
    protected Element render() {
        return column(
            panel(text("Count: " + count).bold().cyan())
                .title("Counter").rounded().fill(),
            panel(text(" ↑/↓: change  q: quit ").dim())
                .rounded().length(3)
        );
    }

    @OnAction(Actions.MOVE_UP)   void inc() { count++; }
    @OnAction(Actions.MOVE_DOWN) void dec() { count--; }

    public static void main(String[] args) throws Exception {
        new CounterApp().run();
    }
}
----

No event loop. No frame rendering. No key parsing. +
Pure state → UI tree.

=== Toolkit: composition

[source,java]
----
@Override
protected Element render() {
    return column(
        header(),
        row(sidebar().length(30), mainContent().fill()),
        statusBar().length(1)
    ).fill();
}

private Element sidebar() {
    return panel(
        list()
            .add(text("Inbox"))
            .add(text("Sent"))
            .highlightSymbol("> ")
            .focusable().id("nav")
    ).title("Folders").rounded();
}
----

Elements are plain Java values. +
`header()`, `sidebar()`, `mainContent()` are just methods returning `Element`.

=== CSS in the terminal

[source,css]
----
/* app.tcss */
.sidebar        { width: 30%; border: solid green; }

#nav:focus      { border-type: double; border-color: cyan; }
#nav:hover      { background: #1e2838; }

.item:selected  { background: #005fd7; color: white; text-style: bold; }
----

[source,java]
----
runner.loadStylesheet("app.tcss");
runner.switchTheme("dark");   // at runtime, live!
----

Pseudo-classes. Themes. No recompile. +
Your designer can edit a `.tcss` file while the app is running.

=== JFR tracing, because it's Java

[source,bash]
----
jcmd <pid> JFR.start name=tamboui \
    settings=profile duration=60s filename=tamboui.jfr
----

Events: `dev.tamboui.terminal.draw`, `dev.tamboui.toolkit.route`, +
focus changes, drag state, event routing… +
Profile your TUI like any other Java app. 🔥
